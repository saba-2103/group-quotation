# API ↔ Schema/Type Drift — 2026-05-14

Snapshot of differences between the live backend (proxied through
`/api/*`) and the local DSL/TypeScript types + schemas. Captured from
real responses on `feat/new-buisiness` against the dev cluster; rerun
the curls below to refresh.

The UI now has two soft mitigations baked in so most drift surfaces as
a graceful fallback rather than a `—`:

- `ColumnConfig.fallbackKey` — DataTable column reads
  `fallbackKey` when its `accessorKey` resolves to a blank.
- `KeyValueField.fallbackKey` — KeyValueGrid field follows the same rule.
- `PageHeader.titleTemplate` — `Quote {quoteNumber|id}` style template
  with `|`-separated fallback chain, sourced from `dataSource`.
- `EditableTable.valueArrayPathFallbacks` — accepts alternative paths
  for the per-row value array (e.g. `planBreakdown` at root in addition
  to `aggregateCensus.planBreakdown`).

The remaining items below are backend fixes — once the backend lands
them, the UI fallbacks become inert and can be removed.

### Census save — round-trip mismatch

`PUT /api/quotation/quotes/:id/aggregate-census` accepts
`{ headcount, planBreakdown[] }` (matches `UpdateAggregateCensusRequest`),
but the subsequent `GET /api/quotation/quotes/:id` returns those values
at the **root** (`headcount`, `planBreakdown`), not nested. This was the
root cause of "headcount resets to 0 after save" — UI was reading
`aggregateCensus.planBreakdown` from a response that no longer has it.

## Action items (suggested priority)

1. **Backend — Quote detail enrichment.** Add `quoteNumber`,
   `clientNumber`, `clientName` to `GET /api/quotation/quotes/:id`.
   Mirrors the summary; removes the need for the UI title fallback.
2. **Backend — Quote aggregate-census round-trip.** Either nest the
   response under `aggregateCensus` (matching the spec + the PUT body)
   or update the spec to reflect the root-level shape. Today's
   asymmetry is silent breakage waiting for the next consumer.
3. **Backend — Proposal detail enrichment.** Same pattern as quote
   (`proposalNumber`, `clientName`, `clientNumber`).
4. **Backend — Policy detail enrichment.** Add `clientName`,
   `clientNumber`. Pick one shape for `estimatedPremium` (Money object
   everywhere, or flat fields everywhere).
5. **Backend — Policy member detail enrichment.** Add
   `policyMemberNumber`, `mobile`, `pendingReason`, `voidReason`,
   `cancellationReason` so the detail page can render reason banners
   without a second fetch.
6. **Spec — `PolicySummaryDto`** is stale; backend already returns the
   extended set. Add `clientName`, `clientNumber`, `effectiveDate`,
   `expiryDate`, `estimatedPremiumAmount`, `estimatedPremiumCurrency`
   or normalize the summary to a strict subset.
