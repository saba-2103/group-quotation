# Keystone UI: New Module Implementation Guide

Welcome to the Keystone UI repository! This guide walks you through implementing a new module or page from scratch.

Keystone UI is heavily **schema-driven**: instead of writing hundreds of lines of React for every screen, we declare layout, forms, and tables in JSON schemas. The React app is an engine that parses those schemas and renders the UI dynamically.

For deeper internals (full widget catalogue, resolver mechanics, action dispatcher), see the [`schema-design-reference/`](schema-design-reference/README.md) docs. This guide is the practical how-to — those docs are the spec.

---

## Worked example: Claims Management

Every step below builds the same fictional "Claims" module. Lock these names in your head — they're reused throughout:

| Concern | Name |
|---|---|

| List endpoint | `GET /api/claims/registrations` |

## 1. Understanding the Schema-Driven Architecture

The pieces:

- **JSON schemas (`/schemas`)** — single source of truth for UI layout. Forms, tables, tabs are all declared here.
- **`WidgetRegistry`** ([`src/components/registry/WidgetRegistry.tsx`](../src/components/registry/WidgetRegistry.tsx)) — maps `type` strings (e.g. `tabs-container`, `data-table`, `form-container`) to React components. If a type isn't in this registry, you get a literal red "Unknown Widget: ..." rendering.
- **`WidgetRenderer`** ([`src/components/registry/WidgetRenderer.tsx`](../src/components/registry/WidgetRenderer.tsx)) — recursive renderer. Takes a `WidgetConfig`, looks the type up, fetches `dataSource` data via [`useSmartQuery`](../src/hooks/useSmartQuery.ts), and renders. It auto-injects `isLoading` and `error` into child props, so most widgets handle loading/error states for free.
- **Schema resolver** ([`src/lib/schemaResolver.ts`](../src/lib/schemaResolver.ts)) — recursively replaces `$ref` pointers. Only two prefixes are supported: `schemas/tabs/...` (dynamic JSON import) and `schemas/forms/...` (resolved through the pre-bundled forms registry).
- **`forms_registry`** ([`schemas/forms/index.ts`](../schemas/forms/index.ts)) — a *generated* TypeScript map that bundles every form schema by id so modals can mount without a network hop. Regenerated by [`scripts/generate_form_index.mjs`](../scripts/generate_form_index.mjs); the `predev` and `prebuild` npm hooks run it automatically.
- **Server Components** — Next.js pages call `resolveSchemaRefs` server-side so the client receives a fully-stitched schema (no hydration mismatches, no client-side loading flashes).

## Step 1: Define the Main Page Schema

1. Create `schemas/claims.json`.
2. The top-level node is **not** a `page` type — there is no `page` widget. The convention is a `stack-layout` with a `page-header` child plus a `tabs-container` (or any other layout) underneath.

```json
{
    "id": "claims-page",
    "type": "stack-layout",
    "props": {
        "className": "p-6 gap-8"
    },
    "children": [
        {
            "id": "claims-header",
            "type": "page-header",
            "props": {
                "title": "Claims Management",
                "description": "Manage claim registrations, assessments, and payouts"
            }
        },
        {
            "id": "claims-tabs",
            "type": "tabs-container",
            "children": [
                { "$ref": "schemas/tabs/claims/registrations.json" },
                { "$ref": "schemas/tabs/claims/assessments.json" }
            ]
        }
    ]
}
```

**Why split tabs into separate files?** Resolution happens server-side, so each tab schema can stay small and reviewable. The resolver explicitly rejects any `$ref` that doesn't start with `schemas/tabs/` or `schemas/forms/` — this is a Webpack/Edge Runtime constraint (see [`schemaResolver.ts`](../src/lib/schemaResolver.ts)).

> **Tabs activate by `id`, not `defaultValue`.** [`TabsContainer`](../src/components/widgets/container/TabsContainer.tsx) seeds the active tab from `children[0].id` and switches via each child's `id`. You may see `props.defaultValue` or `props.value` in older schemas — they are dead config and have no effect today. Order the children so the desired default lands first.

> **`tab-panel` is not a registry-backed widget — it's a structural convention.** `TabsContainer` reads `children[*]` directly and renders each child's `children` inside a Radix `TabsContent`. Using `"type": "tab-panel"` is just a marker for "this child is a tab"; nothing looks it up in `WidgetRegistry`.

## Step 2: Define Tab or Section Schemas

Define each referenced tab. Create `schemas/tabs/claims/registrations.json`.

A tab typically contains a `data-table` to list items plus `headerActions` (e.g. "Register Claim").

```json
{
    "id": "registrations",
    "type": "tab-panel",
    "props": {
        "label": "Claim Registrations",
        "icon": "FileText"
    },
    "children": [
        {
            "id": "registrations-table",
            "type": "data-table",
            "dataSource": {
                "api": {
                    "endpoint": "/api/claims/registrations",
                    "method": "GET"
                }
            },
            "props": {
                "title": "Recent Registrations",
                "rowIdKey": "claimNo",
                "headerActions": [
                    {
                        "id": "register-claim-btn",
                        "label": "Register Claim",
                        "type": "open-modal",
                        "target": "register-claim-form",
                        "variant": "default"
                    }
                ],
                "columns": [
                    { "accessorKey": "claimNo",      "label": "Claim No."    },
                    { "accessorKey": "incidentDate", "label": "Incident Date", "type": "date" },
                    { "accessorKey": "lob",          "label": "LOB" },
                    { "accessorKey": "claimAmount",  "label": "Amount",        "type": "currency", "align": "right" },
                    {
                        "accessorKey": "status", "label": "Status", "type": "badge",
                        "valueMapping": [
                            { "value": "Pending",  "label": "Pending",  "color": "yellow" },
                            { "value": "Approved", "label": "Approved", "color": "green"  }
                        ]
                    }
                ],
                "pagination": { "enabled": true, "pageSize": 20 },
                "rowActions": [
                    {
                        "id": "view-claim",
                        "label": "View",
                        "type": "navigate",
                        "target": "/claims/:id"
                    },
                    {
                        "id": "approve-claim",
                        "label": "Approve",
                        "type": "api-mutation",
                        "visible": { "==": [{ "var": "status" }, "Pending"] },
                        "refreshKey": "/api/claims/registrations",
                        "api": { "endpoint": "/api/claims/registrations/:id/approve", "method": "POST" }
                    }
                ]
            }
        }
    ]
}
```

**Key points:**
- **`tab-panel` is a structural convention** (see callout above). Each tab's `id` (e.g. `"registrations"`) is what the tab bar uses as a value.
- **`headerActions[].type: "open-modal"`** with `target: "register-claim-form"` is how the action system mounts a modal carrying the form schema with that `id`.
- **Columns** require `accessorKey` (the row-data field). `id`, `label`/`header`, and `type` are optional. Only these types have explicit renderers in [`CellRenderer.tsx`](../src/components/widgets/data/CellRenderer.tsx): `number`, `currency`, `date`, `badge` / `status`, `state-badge`, `link`. Anything else (or no type) falls through to plain `String(value)` — that's fine for simple text columns.
- **`valueMapping`** on a `badge` column maps raw values to a label + colour. Without it the badge renders the raw value with the default `outline` variant. Use `state-badge` instead when the column represents a workflow state defined in [`state-map.ts`](../src/components/widgets/state/state-map.ts).
- **`pagination.enabled: true`** is required to show the pager — schemas without it render every row at once.
- **`rowIdKey`** tells the table which row field is the unique id used for `:id` interpolation in row-action URLs.
- **`rowActions[].visible`** uses [JSONLogic](https://jsonlogic.com/). Example: `{"==": [{"var": "status"}, "Pending"]}`. See [`schemas/tabs/accounting/events.json`](../schemas/tabs/accounting/events.json) for richer examples.
- **`refreshKey`** is the endpoint URL whose cached query should be invalidated after the mutation succeeds — usually the same endpoint the table loaded from.

The form referenced by `target: "register-claim-form"` is defined in the next step so it can be pre-bundled and opened without a network hop.

### Schema directories and the `_placeholder.json` files (do not delete)

`src/lib/schemaResolver.ts` resolves `$ref` values via a Webpack dynamic-import template — `await import(\`../../schemas/<dir>/${filename}\`)`. Webpack's static analysis requires **at least one matching `.json` file to exist in each directory at build time**, otherwise the template resolves to `{}` and any `$ref` into that directory silently returns nothing in production (dev usually keeps working from cache).

For that reason, the `schemas/tables/` and `schemas/views/` directories each ship with:

- a `.gitkeep` (so git tracks the otherwise-empty folder), and
- a `_placeholder.json` (so Webpack's chunk manifest has a target).

**Do not delete `_placeholder.json` when "cleaning up empty folders."** Once a real schema lands in either directory, the placeholder can be removed in the same commit. Any new `schemas/<dir>/` directory you reference from `$ref` needs the same treatment.

## Step 3: Create Form Schemas (CRITICAL)

This is where most mistakes happen. The `FormContainer` widget expects a very specific shape.

Create `schemas/forms/register-claim-form.json`.

### The four hard rules
1. **`type` MUST be `"form-container"`.** Do not use `"form"`.
2. **`fields` and `actions` live inside `props`** — not at the root, not in `children`.
3. **Validations use the array form**: `[{ "rule": "required", "message": "..." }]`. Do not use `props.required: true`.
4. **Form submit actions need `type: "api-mutation"`** and `submitAction: true`.

Cross-reference: [`schema-design-reference/06-forms.md`](schema-design-reference/06-forms.md) is the canonical form contract.

```json
{
    "id": "register-claim-form",
    "type": "form-container",
    "props": {
        "title": "Register New Claim",
        "fields": [
            {
                "id": "incidentDate",
                "name": "incidentDate",
                "label": "Date of Incident",
                "type": "date",
                "validations": [
                    { "rule": "required", "message": "Incident date is required" }
                ]
            },
            {
                "id": "claimAmount",
                "name": "claimAmount",
                "label": "Claim Amount",
                "type": "number",
                "validations": [
                    { "rule": "required", "message": "Claim amount is required" },
                    { "rule": "min", "value": 1, "message": "Amount must be greater than 0" }
                ]
            },
            {
                "id": "lob",
                "name": "lob",
                "label": "Line of Business",
                "type": "select",
                "options": [
                    { "label": "Motor",  "value": "MOTOR" },
                    { "label": "Health", "value": "HEALTH" }
                ]
            }
        ],
        "actions": [
            {
                "id": "submit-claim",
                "type": "api-mutation",
                "label": "Submit Claim",
                "submitAction": true,
                "variant": "primary",
                "api": {
                    "endpoint": "/api/claims/registrations",
                    "method": "POST"
                },
                "refreshKey": "/api/claims/registrations",
                "onSuccess": [
                    { "type": "trigger-event", "target": "register-claim-form" }
                ]
            }
        ]
    }
}
```

**The `onSuccess` chain.** Once the submit succeeds, every action in `onSuccess` runs in order. The full vocabulary is in [Reference Tables](#reference-tables). The most common idiom — used in nearly every form in this repo — is a single `trigger-event` targeted at the form's own id, which closes the modal. Combined with `refreshKey`, the parent table re-fetches automatically.

To chain more steps after submit (toast, navigate, patch widget state), append more entries to the array — each runs through the same action dispatcher. See [`schemas/forms/create-event-form.json`](../schemas/forms/create-event-form.json) for the canonical close-modal pattern.

### Common form patterns (not covered above)

The form contract supports more than this guide shows. When you need one of these, jump to the cited section rather than reinventing it:

| Need | Where to look |
|---|---|
| Dynamic select options (fetched from an API) | [`06-forms.md → Dynamic dropdown options`](schema-design-reference/06-forms.md) — `dataSource` on the field with `optionLabel`/`optionValue`. |
| Conditional fields (show/hide based on another field) | [`06-forms.md → Conditional visibility (visibleWhen)`](schema-design-reference/06-forms.md) — JSONLogic against the live form values. |
| Read-only / view-mode form (used on detail pages) | [`06-forms.md → View mode`](schema-design-reference/06-forms.md) — `mode: "view"` on the form or `disabled: true` per field. |
| Edit form pre-filled from the clicked row | [`06-forms.md → Overlaid forms`](schema-design-reference/06-forms.md) — `OverlaidForm` injects the row payload into every field's `defaultValue` automatically; you don't have to do anything beyond using `open-modal` from a row action. |
| Backend validation errors mapped to fields | [`06-forms.md → Backend errors`](schema-design-reference/06-forms.md) — surface `backendErrors[]` from the response envelope. |

## Step 4: Automate Form Registration

When `open-modal` fires, [`OverlaidForm`](../src/components/widgets/forms/OverlaidForm.tsx) fetches `/api/forms/<id>` to load the schema. That route ([`src/app/api/forms/[id]/route.ts`](../src/app/api/forms/[id]/route.ts)) serves from a TypeScript registry instead of the filesystem so it works under Cloudflare's edge runtime, which has no `fs` access.

**CRITICAL**: every time you add or modify a file under `schemas/forms/`, run:

```bash
npm run predev
```

*(or `node scripts/generate_form_index.mjs` directly)*

This script reads every `.json` in `schemas/forms/` and writes [`schemas/forms/index.ts`](../schemas/forms/index.ts), exporting a `forms_registry` map keyed by form id. The `predev` and `prebuild` npm hooks invoke it automatically — but **if you forget and the form isn't in the registry, the API route silently falls back to `dummy-member-form` and the user opens the wrong form with no error.**

`forms_registry` is consumed in two places:
- [`schemaResolver.ts`](../src/lib/schemaResolver.ts) — resolves `"$ref": "schemas/forms/<id>.json"` during server-side schema stitching by looking `<id>` up in the registry.
- [`/api/forms/[id]/route.ts`](../src/app/api/forms/[id]/route.ts) — serves the form schema to `OverlaidForm` at modal-open time.

Neither path reads from disk at request time.

## Step 5: Implement API Routes (Proxies/Mocks)

The schemas above are inert without endpoints. Stand up either a real proxy or a mock catch-all.

Create `src/app/api/claims/[...path]/route.ts`.

```typescript
import { NextRequest, NextResponse } from 'next/server';

export async function GET(
    request: NextRequest,
    { params }: { params: { path: string[] } }
) {
    const path = params.path.join('/');

    // List endpoint
    if (path === 'registrations') {
        return NextResponse.json([
            { claimNo: 'CLM-001', status: 'Pending',  incidentDate: '2026-05-01', claimAmount: 5000  },
            { claimNo: 'CLM-002', status: 'Approved', incidentDate: '2026-04-12', claimAmount: 12000 }
        ]);
    }

    // Detail endpoint — :id resolves to a claimNo because rowIdKey="claimNo".
    const detail = path.match(/^registrations\/([^/]+)$/);
    if (detail) {
        return NextResponse.json({
            claimNo: detail[1], status: 'Pending',
            incidentDate: '2026-05-01', claimAmount: 5000, lob: 'MOTOR'
        });
    }

    return NextResponse.json({ error: 'Not Found' }, { status: 404 });
}

export async function POST() {
    return NextResponse.json({ success: true });
}
```

**DataTable response shape.** DataTable expects a flat array `[{}, {}]`. If your backend returns `{ items: [...], meta: {...} }`, set `dataSource.valueKey: "items"` on the table schema — the `useSmartQuery` hook will read from that key.

## Step 6: Create the Next.js Page

Wire the root schema to a Next.js route.

Create `src/app/claims/page.tsx`. Make it a **Server Component** and resolve `$ref` pointers server-side before handing the schema to the client renderer.

```tsx
import React from 'react';
import { WidgetRenderer } from '@/components/registry/WidgetRenderer';
import { resolveSchemaRefs } from '@/lib/schemaResolver';
import claimsSchema from '../../../schemas/claims.json';

export default async function ClaimsPage() {
    const resolvedSchema = await resolveSchemaRefs(claimsSchema);

    return (
        <div className="w-full h-full bg-background text-foreground">
            <WidgetRenderer config={resolvedSchema} />
        </div>
    );
}
```

[`src/app/accounting/page.tsx`](../src/app/accounting/page.tsx) is the canonical reference — copy its shape.

## Step 7: Detail Routes & Read-Only Views

Most modules have a list page (`/claims`) plus a per-record detail page (`/claims/[id]`). Detail pages typically combine a read-only header (`key-value-grid`), a status display (`state-badge`), and one or more action buttons.

Create `src/app/claims/[id]/page.tsx`:

```tsx
import React from 'react';
import { WidgetRenderer } from '@/components/registry/WidgetRenderer';
import { resolveSchemaRefs } from '@/lib/schemaResolver';
import claimDetailSchema from '../../../../schemas/claims-detail.json';

export default async function ClaimDetailPage({
    params,
}: { params: { id: string } }) {
    const resolved = await resolveSchemaRefs(claimDetailSchema);
    return (
        <div className="w-full h-full bg-background text-foreground">
            <WidgetRenderer config={resolved} />
        </div>
    );
}
```

The detail schema uses `{{id}}` interpolation on its `dataSource.api.endpoint` to pull the route param. The minimal `page.tsx` above won't substitute it on its own — see [`src/app/quotations/[id]/page.tsx`](../src/app/quotations/[id]/page.tsx) for the canonical recipe: after `resolveSchemaRefs`, deep-clone the result with `JSON.parse(JSON.stringify(...))` and walk the tree replacing `{{id}}` in every `dataSource.api.endpoint`. Copy that helper verbatim — it's a few lines and it's the only thing that turns `"/api/claims/registrations/{{id}}"` into the real URL.

**Useful widgets for detail views:**
- `key-value-grid` — read-only label/value pairs (header summary).
- `state-badge` — visual state pill driven by the row's state field.
- `reason-banner` — explanation banner shown when a state has a `reason` set.
- `role-switcher` — gates parts of the UI by the active user role.

These are all in the registry (see [`WidgetRegistry.tsx`](../src/components/registry/WidgetRegistry.tsx)) but rarely needed on the list page — that's why they didn't appear in earlier steps.

## Step 9: Before You Commit

Run these in order:

```bash
npm run predev       # regenerate forms_registry (mandatory if you touched schemas/forms/)
npm run lint
npm run typecheck    # tsc --noEmit; catches WidgetConfig drift early
```

If you added new schema files at the root of `schemas/`, the validator script does a basic shape check (top-level `id`/`type` and any `dataSource.api` block). It does not recurse into `schemas/tabs/` or `schemas/forms/`.

```bash
npx tsx scripts/validate-schemas.ts
```

Then start `npm run dev` and click through the new module: list loads, "Register Claim" modal opens, submit closes the modal and refreshes the table, row "View" navigates to the detail page.

### Field validation rules

Only three rules are wired up today (see [`formContainer/utils.ts`](../src/components/widgets/forms/formContainer/utils.ts)). Unknown rules are silently ignored.

| Rule | Applies to | Example |
|---|---|---|
| `required` | strings | `{ "rule": "required", "message": "Name is required" }` |
| `min` | numbers (value) and strings (length) | `{ "rule": "min", "value": 1, "message": "Must be ≥ 1" }` |
| `max` | numbers (value) and strings (length) | `{ "rule": "max", "value": 100, "message": "Must be ≤ 100" }` |

## Common Gotchas & Troubleshooting

1. **"Unknown Widget: …" in red text.**  
   The `type` you used isn't in [`WidgetRegistry.tsx`](../src/components/registry/WidgetRegistry.tsx). There is no `page` or `tabs-layout` type. Use a registered name.

2. **"My form opens but it's completely blank."**  
   Either `type` is `"form"` (must be `"form-container"`), or `fields`/`actions` are at the root instead of inside `props`.

3. **"Clicked an action button — the wrong form opens (looks like a member form)."**  
   You forgot `npm run predev`. When `/api/forms/<id>` can't find the requested id, it returns `dummy-member-form` with a 200 — no error surfaces. The form must be in [`schemas/forms/index.ts`](../schemas/forms/index.ts). Also check the browser network tab to confirm the data API endpoint exists.

4. **`$ref` failed to resolve.**  
   Path must start with `schemas/tabs/` or `schemas/forms/` — no other prefix works. The resolver attaches `__error` to the failed node — inspect the rendered HTML or server logs.

5. **DataTable is empty but the API returned data.**  
   DataTable expects a flat array. If your endpoint returns `{ items: [...] }`, set `dataSource.valueKey: "items"`.

6. **Tabs render but clicking does nothing / wrong tab is active.**  
   `TabsContainer` keys off each child's `id` and seeds the active tab from `children[0].id`. `defaultValue` / `props.value` are ignored. Reorder children so the desired default is first.

7. **Validation isn't preventing submit.**  
   Use the exact shape `validations: [{ "rule": "required", "message": "..." }]`. Malformed rules are silently ignored.

8. **Hydration error: server-rendered HTML doesn't match client.**  
   You resolved `$ref` pointers in a client `useEffect`. Always resolve in the Server Component (`page.tsx`) and pass the fully-resolved JSON to `WidgetRenderer`.

9. **Row action is always visible / always hidden.**  
   `visible` uses JSONLogic, not the old `{field, operator, value}` shorthand. Example: `{"==": [{"var": "state"}, "PENDING"]}`. Reference: [`schemas/tabs/accounting/events.json`](../schemas/tabs/accounting/events.json).

10. **Modal closes but table doesn't refresh.**  
    Add `"refreshKey": "<endpoint>"` to the submit action. After the mutation succeeds [`useActionHandler`](../src/hooks/useActionHandler.ts) invalidates every query whose first queryKey element `startsWith(refreshKey)` — and the queryKey's first element is the table's `dataSource.api.endpoint` (without query-string). So `refreshKey: "/api/claims/registrations"` matches the table's endpoint as a prefix; you do NOT need to append `?page=…`.
