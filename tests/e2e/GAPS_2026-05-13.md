## GAP 1 — DataTable never renders its `title` prop

**Severity:** demo-blocking for the workbench framing PROP-0009 introduced.
The Inbox is rendered as a single "Inbox" heading above multiple unlabeled
tables. A demoer switching role can see queues, but the rows above them have
no header explaining what they are looking at. Same impact on every list
page (Quotations, Proposals, Policies, Clients).

**Failing tests (10):**
- `dashboard-inbox.spec.ts` — every `role=X sees section "Y"` test (7 of them)
- `dashboard-inbox.spec.ts` — "Sales sees BOTH inbox sections..."
- `dashboard-inbox.spec.ts` — "a role only sees their own inbox section — no cross-leak"
- `demo-narrative.spec.ts` — "Ops sees REPAIR_PENDING members in their Inbox"

**Root cause:** [src/components/widgets/data/DataTable/index.tsx:188-200](../../src/components/widgets/data/DataTable/index.tsx) renders the toolbar with only `headerActions` and the Export dropdown — never the `title` prop. The only place `title` shows is the empty-state. Every schema passes `title: "..."` to `data-table` (verified in [schemas/dashboard.json](../../schemas/dashboard.json), [schemas/quote.json](../../schemas/quote.json), [schemas/proposal.json](../../schemas/proposal.json), [schemas/policy.json](../../schemas/policy.json)) and gets silently dropped.

**Fix (small, scoped):** in `DataTable/index.tsx` toolbar, render the title to the left of headerActions:

```tsx
<div className="flex items-center justify-between px-3 py-2 border-b gap-2">
  {title && <h3 className="text-sm font-semibold text-foreground">{title}</h3>}
  <div className="flex items-center gap-2">
    {headerActions?.map(...)}
    <DropdownMenu>... Export ...</DropdownMenu>
  </div>
</div>
```

`title` is already destructured from props in the component — just needs to be rendered. ~5 LOC.

## GAP 2 — UW/Member/Ops Inbox row click lands on PAM Member detail with no policy-member actions

**Severity:** demo-blocking for §3 (UW review) and §4 (Ops repair). The
member-self-confirm flow (§2B) has the same problem.

**Failing tests (4):**
- `demo-narrative.spec.ts` — "UW can open a REFERRED_TO_UW member and see UW approve/reject"
- `demo-narrative.spec.ts` — "Ops can open a REPAIR_PENDING member detail and see Edit & re-classify"
- `detail-pages.spec.ts` — "UW viewing a REFERRED_TO_UW member sees UW approve + UW reject"
- `detail-pages.spec.ts` — "Ops viewing a REPAIR_PENDING member sees Edit & re-classify"

**Root cause:** The Inbox sections for `uw`, `member`, and `ops` use
`linkRoute: "/policy-admin/members/by-policy-member/:id"` ([schemas/dashboard.json](../../schemas/dashboard.json)).
That route is [src/app/policy-admin/members/by-policy-member/[policyMemberId]/page.tsx](../../src/app/policy-admin/members/by-policy-member/[policyMemberId]/page.tsx) —
**a redirect page**, not a render page. It resolves `policyMemberId → memberId`
and redirects to `/policy-admin/members/:id`, which uses `schemas/member-detail.json`
(the PAM-side member view). `member-detail.json` has no `roleActions` /
`stateActions` for the policy-member workflow — no `uw-approve`, no
`uw-reject`, no `repair-edit`, no `confirm-maf`. So clicking from the
Inbox loses the role's actionable surface.

**Fix options:**

**(a) Add a standalone PolicyMember detail route** — `src/app/issuance/policy-members/[policyMemberId]/page.tsx` — that loads `schemas/policy-member-detail.json` directly without needing a proposal id in the URL. ~30 LOC mirroring the existing `[id]/members/[memberId]/page.tsx` pattern but without the proposal nesting. Then update the dashboard Inbox `linkRoute` for the 3 affected sections.

**(b) Update the Inbox row data to include `proposalId`** and use the existing nested route `/issuance/proposals/:proposalId/members/:id`. Cheaper but requires the search endpoint response shape to include `proposalId` per row.

**(c) Add the policy-member actions to `member-detail.json`** so the redirect path stays. Bad — `member-detail.json` is the PAM-side surface; mixing policy-member workflow actions onto it muddies the schema.

Recommend **(a)**. Mechanical, schema is reusable, no new actions, no shape changes.

## What works (positive findings)

- **All 6 narrative roles** are selectable in the switcher; default = Sales.
- **Server-side menu filter** (`/api/config/app?role=`) returns the right
  menu shape for every role — all 12 menu-filter tests pass (6 API + 6 UI).
- **Action-level RBAC** on Quote detail works: Sales sees their actions, MPH
  doesn't see Sales actions, Sales doesn't see MPH-only Accept/Reject.
- **No console errors** on any of the 6 reachable routes for the Sales role,
  and no console errors when switching to any of the other 5 roles on the
  dashboard.
- **Backend proxy reachable** for all 6 search endpoints we use in tests.
- **No stale "maker/checker/viewer" copy** anywhere on the live dashboard or
  quote detail.
- **localStorage role persistence + invalid-stored-role fallback** both work.
- **List pages** (quotations, proposals, policies, clients, member-quotes):
  every page loads with header + data area + Filter trigger + first-row
  navigation.
