## GAP 0 — Schema incoherence: Sales holds BOTH sides of the maker-checker overlay

**Failing test (1):** `tests/e2e/schema-coherence.spec.ts` — *quote-detail.json: no role holds both sides of a two-sided handshake*

> Error: schema quote-detail.json: role "sales" holds BOTH sides of the
> maker-checker approval overlay (send-for-approval + submit). One of them
> must belong to a different role, or both must be removed.

**Why it matters:** the demo persona Sales is now both the submitter and the
approver of the same quote. The role-asymmetric "send-for-approval → MPH
approves" theatre that was preserved in CORE_MEMORY is gone — a single role
holds both halves of the handshake, which makes no operational sense.

**Where it came from:** PROP-0009's schema sweep mapped both `maker` and
most of `checker` to `sales`, including the maker-checker overlay actions
that were always paired.

**Fix options:**
- **(a) Drop the overlay scaffolding.** Strip `send-for-approval`,
  `clear-approval`, and `submit`'s coupling from `quote-detail.json`.
  Simplest. Demo doesn't show internal QA anyway.
- **(b) Move `submit` to `mph`** as the "approve & forward" step paired with
  Sales's `send-for-approval`. Keeps the asymmetry but conflates internal
  approval with cross-org acceptance, since MPH also already has accept/reject.

Recommend **(a)** — the demo narrative doesn't include the internal QA
beat. Clean break.

## GAP 3 — `useRole must be used inside <RoleProvider>` crash when widgets render via portals

**Failing tests (13):**
- `interactions.spec.ts` — Sales clicking "New Quote" → modal failed to mount
- `interactions.spec.ts` — Sales clicking "Add member" → page errors
- `interactions.spec.ts` — Quote detail tab clicks → page errors
- `interactions.spec.ts` — Dashboard inbox row click (all 6 roles) → request aborted
- `interactions.spec.ts` — Sales menu navigation → request aborted
- `interactions.spec.ts` — Filter dropdown open (all 3 list pages) → request aborted

**Pageerror captured by `tests/e2e/_repro-newquote.spec.ts` (since deleted):**

```
pageerror: useRole must be used inside <RoleProvider>
  at WidgetRenderer (...)
```

**Root cause:** PROP-0009 added `const { role } = useRole();` at the top of
`WidgetRenderer.tsx`. The hook throws when context is missing
([src/hooks/useRole.ts:9-11](../../src/hooks/useRole.ts)). When a schema
widget renders inside a portal — Radix Dialog content, Dropdown menus,
Toaster — and the React subtree mounts outside the `<RoleProvider>` wrapper
in `layout.tsx:39`, the inner `WidgetRenderer` throws and crashes the page.

This is the bug the user reported by clicking "New Quote" and watching the
system crash.

**Severity: demo-blocking.** Every form modal, every action that opens a
dialog, every Radix dropdown that mounts schema-rendered widgets is broken.

**Fix:** Make `useRole()` resilient — return `DEFAULT_ROLE` when context is
missing instead of throwing. ~3 LOC in `src/hooks/useRole.ts`. The
role-aware behavior still works correctly when actually inside the provider
(99% of cases). The fallback only kicks in for portal-mounted widgets that
shouldn't depend on role anyway.

Alternative: move `<Toaster />` (and any global modal mount points) INSIDE
`<RoleProvider>` in `layout.tsx`. Doesn't fully solve the issue — Radix's
own internal mount points are still outside our component tree.

The hook-resilience fix is the right call.

## GAP 1 — DataTable never renders its `title` prop

**Failing tests (10):**
- `dashboard-inbox.spec.ts` — every `role=X sees section "Y"` test (7 of them)
- `dashboard-inbox.spec.ts` — "Sales sees BOTH inbox sections..."
- `dashboard-inbox.spec.ts` — "a role only sees their own inbox section — no cross-leak"
- `demo-narrative.spec.ts` — "Ops sees REPAIR_PENDING members in their Inbox"

**Root cause:** [`DataTable/index.tsx:188-200`](../../src/components/widgets/data/DataTable/index.tsx) — the toolbar renders `headerActions` and the Export dropdown but never the `title` prop. The only place `title` shows is the empty-state at line 135. Every list schema passes `title: "..."` and gets silently dropped.

**Severity: demo-blocking for the workbench framing.** The Inbox is rendered as a single "Inbox" heading above multiple unlabeled tables. Demoers switching roles see queues with no headers explaining what they are.

**Fix:** ~5 LOC in `DataTable/index.tsx` toolbar — render `{title && <h3>...</h3>}` to the left of `headerActions`.

## GAP 2 — UW/Member/Ops Inbox row click lands on PAM Member detail with no policy-member actions

**Failing tests (4):**
- `demo-narrative.spec.ts` — UW can open a REFERRED_TO_UW member and see UW approve/reject
- `demo-narrative.spec.ts` — Ops can open a REPAIR_PENDING member detail and see Edit & re-classify
- `detail-pages.spec.ts` — UW viewing a REFERRED_TO_UW member sees UW approve + UW reject
- `detail-pages.spec.ts` — Ops viewing a REPAIR_PENDING member sees Edit & re-classify

**Root cause:** Inbox `linkRoute: "/policy-admin/members/by-policy-member/:id"` ([`schemas/dashboard.json`](../../schemas/dashboard.json)) is a redirect page ([`src/app/policy-admin/members/by-policy-member/[policyMemberId]/page.tsx`](../../src/app/policy-admin/members/by-policy-member/[policyMemberId]/page.tsx)) that forwards to `/policy-admin/members/:id` — which uses `schemas/member-detail.json`. That schema has no policy-member workflow actions. So UW/Member/Ops can SEE their queues but the click-through strips their action bar.

**Severity: demo-blocking for §3 (UW review) and §4 (Ops repair).** Member's §2B confirm-MAF flow likely also affected.

**Fix:** Add `src/app/issuance/policy-members/[policyMemberId]/page.tsx` that loads `policy-member-detail.json` directly (~30 LOC mirroring the existing nested route). Update the 3 affected Inbox `linkRoute`s.
