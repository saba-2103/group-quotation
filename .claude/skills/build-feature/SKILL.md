# Build-Feature Orchestrator

Run a verb-driven, multi-stage pipeline that turns a feature ask — fuzzy or specific, fresh or proposal-backed — into a clarified, designed, built, verified, and locally-previewable change. The skill **stops at user gates** between CLARIFY → DESIGN → BUILD; it does not silently advance past ambiguity.

This skill **subsumes the input-source distinction** between a free-form ask and an approved proposal. The same pipeline runs for both; the CLARIFY stage adapts to how complete the input is.

## Inputs
- `$ARGUMENTS` — required. May be:
  - a proposal id (e.g. `PROP-0001`) — load and analyze the proposal first
  - a free-form ask (e.g. `"add a wizard for Add Member"`)
  - a path to a design or feature note (e.g. `docs/notes/foo.md`)
  - `--no-preview` — skip the local preview at SHIP; stop after VERIFY
  - `--dry-run` — stop after DESIGN approval; do not BUILD

## Source-of-truth locations (verify before using)
- Handoff entry point: `context/HANDOFF.md`
- Core memory: `context/CORE_MEMORY.md`
- Proposals: `proposals/PROP-*.md`
- Existing skills (for handoff or reference): `.claude/skills/*/SKILL.md`
- Implementation guide: `docs/NEW_MODULE_IMPLEMENTATION_GUIDE.md`
- Primitives: `src/components/ui/*.tsx`
- Composed widgets: `src/components/widgets/{container,controls,data,forms,items,layout}/`
- Mocks / config: `src/mocks/`
- Logs (create per-run): `agent_logs/build-feature/<run-id>/`
- Design docs (create per-run): `context/build-feature/<run-id>/design.md`

Run-id format: `<YYYY-MM-DD>-<short-slug>` derived from the ask (e.g. `2026-05-01-wizard-widget`).

## Pipeline

### 1. DISCOVER — understand the ask in context
- Read `context/HANDOFF.md` first when it exists, then follow its links before relying on conversation context.
- Read `context/CORE_MEMORY.md` first and treat it as a standing execution preference unless the user overrides it in this run.
- Read `agent_logs/usecases.json` (if generated by `extract-usecase`) to ensure the feature adheres strictly to the defined business rules and domain logic.
- If input is `PROP-NNNN`: read the proposal file in full. Note Problem, Proposed change, Recommendation, and any open questions.
- If input is a free-form ask: echo back your understanding in one short paragraph so misunderstandings surface immediately.
- Read `docs/NEW_MODULE_IMPLEMENTATION_GUIDE.md` when the ask involves a new page, module, schema, widget, form, or API surface.
- Survey the codebase for related/adjacent code: similar features, existing primitives, sibling files, naming conventions, test patterns.
- Identify three lists: **known** (locked decisions), **ambiguous** (needs user input), **derivable** (can be inferred from existing patterns — note the source).
- Output: `agent_logs/build-feature/<run-id>/discover.log` — input source, hashes, surveyed paths, the three lists.

### 2. CLARIFY — close the gaps with the user
- Present 3–5 sharp questions per round, not a wall. Focus on items where guessing would mislead the design.
- After asking, **STOP**. Do not advance until the user answers.
- Integrate answers, then ask another round if needed. Loop until the ambiguous list is empty or stable.
- If input was a fully-fleshed proposal with no opens: do a single confirmation round ("Here's what I read; anything to add?") and proceed.
- If a question is decidable from existing patterns, propose the answer with reasoning rather than asking ("I'll follow the FilterBar precedent unless you'd rather X — okay?").
- Output: `agent_logs/build-feature/<run-id>/clarify.log` — append each round (questions, answers, decisions).

### 3. DESIGN — write the doc, get explicit approval
- Write `context/build-feature/<run-id>/design.md` with: Goal, In-scope, Out-of-scope, Data shape, Components/files to add or modify, API surface, Edge cases, Test plan.
- For UI work, include a **Design Preview** section before BUILD starts. It should show the planned composition shape (widgets/layouts/routes/schema snippets), note desktop/mobile behavior, and make it obvious whether the solution is composed from existing primitives or requires a new component/layout layer.
- If the feature must use an interim pattern ahead of a planned future architecture, add an **Architecture Transition Note** section: what the interim contract is, what risk it carries, and what should simplify once the future architecture lands.
- When there is a choice between a narrow demo slice and modest upfront foundational work, bias the design toward the option that reduces downstream rework and gives later dependent proposals a cleaner base. Call out the added upfront cost explicitly.
- Show the design to the user (or summarize it inline) and ask for **explicit approval**.
- If the user requests revisions, edit the doc and re-confirm. If they reject the direction, return to CLARIFY.
- **Do not write code until the user approves.** This is a hard gate.
- Output: `agent_logs/build-feature/<run-id>/design.log` — link to design.md, approval timestamp.

### 4. BUILD — implement on a sane branch
- **Branch hygiene check first** (mandatory):
  - Run `git status` and `git branch --show-current`.
  - If working tree is clean **and** current branch is not `main`/`master`: build on the current branch.
  - Otherwise (dirty tree, or on main/master): create `feature/<run-id>` from current HEAD and switch to it before any edits.
  - Log the decision and the branch used.
- Implement per the approved design. Prefer existing primitives, but compose feature-local workflow UI where needed. Do not hallucinate shared design-system components that do not exist.
- Follow `docs/NEW_MODULE_IMPLEMENTATION_GUIDE.md` for schema-driven pages, server-component page setup, form registration, API route expectations, and navigation integration when those concerns apply.
- If the feature clearly needs a new widget, you MAY build one instead of shrinking scope. The widget must be designed as a real reusable asset, not a one-off patch.
- A new widget must include appropriate registry integration, type/config support, focused tests, and at least one real consumer in the feature being built.
- Every visible action added in this run must terminate in a real route, modal, or endpoint. Do not leave dead-end buttons in the UI.
- Use `TodoWrite` to track per-file/per-component progress; mark items done as you finish them.
- Commit incrementally with clear messages. Don't bundle unrelated changes.
- If a need arises that wasn't in the design, **stop and surface it** instead of expanding scope.
- Output: `agent_logs/build-feature/<run-id>/build.log` — branch_used, files_changed, components_used, commits, scope_changes.

### 5. VERIFY — prove it works
- Run available gates: `npm run lint`, type-check (via `tsc --noEmit` if available), `npm test` (or the relevant subset).
- Self-review: re-read each new file. Verify imports resolve, props match the component's source interface, edge cases from the design are handled.
- Verify route/action integrity: every new navigation target exists, every endpoint used by new UI exists, and dynamic detail routes actually consume params.
- For workflow features, add or update interaction tests that cover at least one end-to-end happy path and one critical negative guard from the design. Rendering-only tests are not enough for approvals, uploads, LWD exits, claim initiation, renewals, or role-specific actions.
- If something fails: fix in place, re-run, re-log. If you can't fix it, stop and ask — do not paper over.
- Output: `agent_logs/build-feature/<run-id>/verify.log` — each gate's pass/fail, what was fixed, residual risks.

### 6. SHIP — local preview (no public deploy)
- Run `npm run preview` (the OpenNext Cloudflare build + local server on `http://localhost:8787`).
- Wait for the build to succeed; report the URL to the user.
- Tell the user: *"Preview running locally. Run `/preview-and-deploy` if you want to deploy publicly."*
- **Do not auto-deploy.** Public deploy is the user's call.
- If `--no-preview` was passed, skip this stage entirely — do not write a ship.log.
- Output: `agent_logs/build-feature/<run-id>/ship.log` — build pass/fail, preview URL, caveats. Only written if the stage actually ran.

## Operational constraints

- **Clarify before coding.** If a critical question is open, STOP and ask. Do not guess.
- **One feature per run.** If scope creeps in CLARIFY or DESIGN, propose splitting into multiple runs (and optionally file follow-up `/propose` items).
- **Prefer strategic front-loading over throwaway slices.** If a modest expansion inside the same feature meaningfully reduces future churn, integration risk, or repeated rework, prefer the more reusable design and surface the tradeoff to the user instead of automatically shrinking scope.
- **Respect composability boundaries.** Schemas are meant to compose widgets and layouts. Do not turn a workflow widget into a generic layout engine just to squeeze in a consumer; if the existing composition model is insufficient, stop and either design a proper new layout/widget or revise the composition plan.
- **Prefer event/state contracts over schema callbacks.** If composed child widgets need to affect workflow completion, document a small widget-state or eventing contract rather than smuggling runtime callback concepts into schema.
- **Hard gates are hard.** CLARIFY ends only when the user signals so; DESIGN ends only with explicit approval. Don't infer consent from silence.
- **Branch hygiene is non-negotiable.** Never edit code on a dirty tree or on `main`. Run the check at the top of BUILD.
- **No faked stages.** Do not write a stage's log until that stage has actually run. Inherits the rule from `specs-to-draft` L-007.
- **Prefer composition first, but build a new widget when clearly required.** Do not reduce scope just because the library is incomplete. If a new widget is the right answer, build it to repo quality.
- **No half-built widgets.** New widgets must have an explicit API/config contract, accessibility-conscious behavior, focused tests, and proper registry wiring where applicable.
- **No new top-level deps without asking.** Prefer what's already in `package.json`.
- **Test coverage proportional to risk.** Unit tests for new branching logic, plus interaction tests for workflow features and route/action integrity where user-visible journeys are involved. Note coverage gaps in verify.log so they're visible.
- **Update proposal status.** If input was `PROP-NNNN`, update the proposal's `status` to `in-progress` at BUILD start and `done` after the PR is opened (or `awaiting-deploy` after preview if no PR).
- **Don't bypass the user.** This skill exists because the ask was fuzzy; the user is part of the loop. Surface decisions, don't just take them.

## Kickoff

1. Resolve input:
   - If `PROP-NNNN`: read `proposals/PROP-NNNN-*.md` and capture the recommendation.
   - If free-form: echo back the ask in your own words so the user can correct early.
2. Generate `run-id` and create `agent_logs/build-feature/<run-id>/` and `context/build-feature/<run-id>/`.
3. Begin DISCOVER. Proceed through the pipeline, logging after each stage.
4. End with a short report: what was built, on which branch, preview URL (or "not previewed"), and what's left for the user to do (review the PR, run `/preview-and-deploy`, etc.).

## Integration with build-backend

If the feature requires a real backend implementation (and the user explicitly asks for the backend to be built instead of mocked), hand off the backend generation to `/build-backend`. Do not attempt to build a real Python backend within `/build-feature`. `/build-backend` is specifically designed to run autonomously and generate production-ready backend code based on the frontend schemas and proposals.

## When to use this vs `/execute-proposal` (decision rule)

Default to `/build-feature`. Reach for `/execute-proposal` only when **all** of the following hold:

- A `proposals/PROP-NNNN-*.md` exists with `status: approved`.
- The proposal's design section is fully locked — no open questions, no ambiguity that would benefit from CLARIFY.
- The change is mechanical enough that going through CLARIFY → DESIGN gates would be a no-op.

Anything else — fresh asks, partially-scoped proposals, anything where the design might evolve under questioning — goes through `/build-feature`. The same rule appears in `context/HANDOFF.md` Skill catalogue so callers don't have to read this file to decide.

`specs-to-draft` remains separate: it scaffolds an entire PRD's UI surface in one autonomous pass with minimal user gating. `build-feature` is the opposite shape — one feature, lots of user gates.

## Integration with specs-to-tests

`/specs-to-tests` can hand off implementation bugs to `/build-feature` automatically. When the input is a structured bug handoff (containing schema path, expected values, actual values, and root cause hypothesis):

- **CLARIFY is streamlined** — the bug description is already narrowed to a concrete workflow, route, role, or render gap.
- **DISCOVER** should determine whether the failure is a route/action integrity bug, workflow/state bug, authorization bug, or widget/render bug.
- **DESIGN** may be a widget fix, a route build-out, a workflow implementation, a role/guard correction, or a newly required widget depending on the failure.
- After BUILD + VERIFY, re-run the specs-to-tests suite to confirm the fix resolved the contract violation.
