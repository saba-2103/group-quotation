### Track F — ActionBar `stateField` override

**Files:**
- `src/components/widgets/actions/ActionBar.tsx` — `ActionBarPropsResolved` adds `stateField?: string` (default `"state"`); state-resolution reads `liveEntity[stateField]` / `fetchedEntity[stateField]` instead of hardcoded `.state`.
- `schemas/quote-detail.json` — main action-bar declares `"stateField": "status"`.
- `schemas/member-quote-detail.json` — same.

**Why this surfaced now:** the original ActionBar was written assuming every entity uses `state`. Quote and MemberQuote DTOs use `status` (DSL canon — `Quote.status` and `MemberQuote.status`). Result: state-gating computed against `entity.state = undefined`, every action fell into the "not in this state" tooltip path. Mock fixtures and proxy-mode responses both have this — the bug was always there, just nobody clicked the disabled buttons.

### Track G — Proposal-scoped /members proxy-aware fallback

**Files:**
- `src/app/api/issuance/[[...path]]/route.ts` — adds `findProposalProxyAware()` (consults local store first, falls back to backend fetch) and `forwardToBackend()` (issues a request to the live backend and returns the response). The GET + POST handlers for `proposals/:proposalId/members` use both: in proxy mode, look up the proposal upstream, get its `policyId`, then forward to backend's `/api/issuance/policies/:policyId/members` (which IS DSL canon).

**Why this surfaced now:** the local mock route's `findProposal()` only consults the in-memory mock store. In proxy mode, real backend proposals (UUIDs) are never in that store — every lookup 404'd. Backend's proposal-scoped /members endpoint is stale OpenAPI-only (returns 404 live), so simply removing the MOCK_ONLY pattern wouldn't help; the schema needs to keep using the proposal-scoped shortcut while the route translates to policy-scoped against backend.

### Note on Issue 1

User reported quote detail tabs (Plans / Census / Member-mapping) show new field labels but values render as "—". Investigated:

```
$ curl /api/quotation/quotes/<empty-quote>
{ "plans": [], "censusFileFormatJson": null, "memberToPlanMappingJson": null, ... }

$ curl /api/quotation/quotes/<seeded-quote>
{ "plans": [{...}], "censusFileFormatJson": "{...}", "memberToPlanMappingJson": "{...}", ... }
```

Most pre-existing dev backend quotes are skeletal — only the 9 we seeded yesterday have full plans/census/mapping. "—" is the correct rendering for empty fields. Demo should drive a seeded quote (e.g. `1808cd1c-…`) — those populate correctly.

If we want to encourage the demo to land on a populated quote, options:
- Sort the quote list so seeded ones rise to the top (would need backend support or client-side sort by populated-fields).
- Filter the list to exclude empty quotes (would need a "decorated only" toggle).
- Demo doc already names QTE-2026-0002 which is a mock-mode fixture; in proxy mode the equivalent is whichever seeded quote has plans. Worth flagging in the demo script.
